& the minimal layout engine

How a React <Scroll> element becomes a clipped, focusable, D-pad-scrollable viewport on a PlayStation Vita — without a layout engine, except for the one tiny rule it ships with: stack children in a column.

01

Two threads, one tree

VitaDeck renders React on a QuickJS thread and draws with raylib on the UI thread. The two meet at a double-buffered instance tree: JS mutates a back buffer, then swaps an immutable snapshot to the front. Scroll position deliberately lives on neither side of that handoff — it is UI-thread state keyed by instance id, so scrolling never round-trips through React.

02

The minimal layout engine

Every other VitaDeck element is absolutely positioned. Scroll introduces exactly one layout behavior — column flow — implemented in ~20 lines (scroll_flow_step). Direct Rect and Button children participate; anything else renders at its own coordinates and is skipped by the flow.

y is a margin, not a position

A stacked child's y nudges it down from its flow slot and pushes every later sibling — exactly like margin-top.

x is free

x is relative to the padded content area, untouched by the flow. That single degree of freedom is what lets chat bubbles alternate left / right.

Heights must be explicit

The flow reads height straight from props — no text measuring. That keeps hit-testing font-free and the whole engine trivially small.

// scroll_flow_step, per child:
base    = flow_y
place_y = base + child.y
flow_y  = place_y + child.height + gap

// after the loop:
content_height = flow_y − gap + 2·padding
max_scroll     = max(0, content_height − viewport_height)
03

Drawing one frame

Rendering re-runs the same flow math, offsets everything by the stored scroll position, and lets the GPU do the overflow work via raylib's scissor test. The offset is clamped against max_scroll every frame, so content that shrinks can never strand the viewport in empty space. 

// src/ui/render.c — render_scroll_instance (shape of it)
offset = clamp(scroll_get_offset(id), 0, max_scroll);   // self-healing

BeginScissorMode(abs_x, abs_y, width, height);          // ✂ overflow:hidden
content_origin = (abs_x + padding, abs_y + padding - offset);
for each child:
    if scroll_flow_step(child, gap, &flow_y, &base)   // rect / button → stacked
        render(child at content_origin + base)
    else
        render(child at content_origin)               // text etc → free placement
EndScissorMode();

if (max_scroll > 0)  draw_scrollbar(thumb ∝ viewport / content);
if (focused(id))     draw_focus_ring();                // white 2px outline

Hit-testing in instance_tree.c mirrors this exactly: it walks the same flow, subtracts the same offset, and refuses to hit children outside the viewport — so a button scrolled out of view can't be clicked through the clip edge.

04

Focus, then scroll

The container registers itself as a single focusable element (its children don't compete). Once focused, it consumes up/down — navigation only wins back the D-pad when the list is already at an edge. One press at the limit moves focus away; anywhere else, it scrolls.

InputBehaviorSpeed
D-pad / arrow keys held scrolls continuously while the container has focus; consumed until the edge is reached, then the next press navigates focus away. 8 px / frame ≈ 480 px/s
Left analog stick Proportional smooth scrolling past a 25% deadzone — tilt gently for slow reading, slam it to fly. ±14 px / frame max
Mouse wheel No focus needed: instance_scroll_at(x, y) finds the deepest scroll container under the cursor and nudges it. 40 px / notch
Runtime restart The Shell swapping Deck Apps calls scroll_reset() — stale offsets from the previous app's ids are dropped wholesale. —
