Host control companion — starter commands (iteration 0)

Working doc for a Host Control Companion: a program on the desktop that accepts structured requests from Vita Deck Apps (via VitaDeck) to control the machine. VitaDeck’s glossary uses host elsewhere (macOS dev target, React host elements); this doc uses Host Control Companion for this app to avoid collisions.

Goal for the first slice

Prove end-to-end delivery of a small, typed command set from a Deck App to the desktop: connect, verify, then perform low-risk OS actions that do not require typing arbitrary text or driving the cursor (defer those until the threat model tightens).

Initial commands

Use each row’s command string as the JSON command field in POST /v1/command; omit or empty payload unless noted.

Command Purpose Notes
host.capabilities Handshake: protocol version, supported commands, optional host display name. Deck App can adapt UI (grey out unsupported actions). Pair with a Deck-side deck.ready if you want symmetry.
host.echo Round-trip test with opaque payload + server timestamp. Validates wiring before anything touches the OS; keep payloads size-capped.
host.power.sleep_displays Turn displays off / sleep displays only (not full suspend if avoidable). Useful “remote control” moment; scope tightly so it is not mistaken for shutdown. macOS: pmset displaysleepnow.
Deferred (good second iteration):
  • Mouse/keyboard injection, clipboard read/write, arbitrary shell — high abuse surface.
  • Launching apps or URLs unless paired with an explicit allowlist model.

Transport (decided)

  • Network: Same Wi‑Fi LAN as the Vita.
  • Discovery / addressing: Operator-entered IPs only — on the Vita, configure the PC’s Host Control Console Address in VitaDeck Shell; on the companion, optionally configure the Vita’s LAN IP when an implementation needs it (callbacks, allowlists, or logging). No mDNS, tokens, or pairing ceremony in this iteration (Host Control Trusted LAN).
  • Wire format: HTTP request/response — JSON (Content-Type: application/json), no WebSocket.
  • Routing: Single dispatch endpoint — POST /v1/command with body { "command": string, "payload"?: object }; responses JSON only.

Response envelope (decided)

All POST /v1/command responses use VitaDeck LAN JSON Result (same family as Runtime Upload POST): Content-Type: application/json.

  • Success: { "ok": true, … } — extra keys depend on command (e.g. capabilities echo metadata).
  • Failure: { "ok": false, "code": string, "message": string } — HTTP status carries error class (malformed body, unknown command, unsupported OS action, etc.).

When you implement: mirror this envelope in shared TypeScript types for Deck Apps and the companion (not part of this planning-only iteration).

Cross-cutting requirements (still open)

  • Errors: failures always use VitaDeck LAN JSON Result (ok: false, code, message).
  • Idempotency: repeated host.power.sleep_displays is safe (displays already off).

Locked API appendix

The companion listens on all interfaces, starting at TCP port 8797 and falling forward through 8806 if ports are busy. Deck Apps send JSON to POST /v1/command: 

{ "command": "host.echo", "payload": { "value": "hello" } }

Failures use the same JSON envelope. Transport, routing, and command-dispatch codes:

Status Code Meaning
400malformed_requestBody is not valid JSON or does not contain a string command.
400unknown_commandThe command string is not a registered Host Control command (not in the protocol command set).
404not_foundPath is not part of the Host Control Companion HTTP Contract.
405method_not_allowedThe endpoint exists but does not accept this method.
413payload_too_largeRequest body exceeds the companion's JSON request cap.
415unsupported_content_typeRequest is not application/json.
422invalid_payloadThe command is known, but its payload is invalid.
501unsupported_platformThe command is not implemented on the current desktop OS.
501command_failedA command supported on this platform failed while executing (e.g. the OS tool returned an error).

command_failed message carries the runtime error detail when available.

Command successes:

  • host.capabilities: { "ok": true, "protocolVersion": 1, "hostName": string, "platform": string, "commands": string[] }
  • host.echo: { "ok": true, "payload": object, "receivedAt": string }; echo payloads are capped by the request limit.
  • host.power.sleep_displays: { "ok": true } when the companion successfully asks the OS to sleep displays.

Open alignment with VitaDeck

Runtime Upload is desktop → Vita over HTTP at Runtime Upload URL. Host Control Companion is Vita → desktop over LAN using Host Control Console Address — inverse direction, separate surface (both documented in CONTEXT.md).
