# VitaDeck Deck App Packages Plan

## Goal

Decouple Deck App authoring and packaging from the VitaDeck runtime build. A Deck App author should be able to use a local npm package, build a portable Deck App Package Directory, and have VitaDeck consume that package as its active Deck App.

This is the next iteration after source-selected in-repo Deck Apps. Runtime upload is still future work, but the artifact shape should be suitable for later upload.

## Iteration 2 Choices

- The author-facing npm package is the **VitaDeck SDK**, published locally as `@vitadeck/sdk`.
- The **VitaDeck SDK** exposes the **VitaDeck Runtime API** and initial Deck App build tooling.
- Deck App authors write normal TypeScript and ES modules.
- A Deck App source entry default-exports a React component; authors do not call registration or render APIs.
- The SDK build generates the runtime-loadable **Deck App Package Entry**.
- The built user-provided unit is a **Deck App Package**.
- The initial package artifact is an unpacked **Deck App Package Directory** with a `.vdapp` suffix.
- Future Runtime Upload may zip the `.vdapp` directory for transport without changing the package contents.
- Every `.vdapp` directory contains a versioned `manifest.json`.
- VitaDeck consumes an already-built `.vdapp` directory, not Deck App source metadata.
- VitaDeck owns the runtime bootstrap, native bindings, reconciler, theme provider, and React instance.
- React is a **Runtime Dependency** and must not be bundled into Deck App Packages.
- The SDK declares React compatibility through peer dependencies and validates the resolved React version during builds.
- The SDK provides a reusable TypeScript config preset for Deck App projects.
- The SDK provides an author-facing scaffold command for new Deck App projects.
- The JavaScript codebase becomes a pnpm workspace rooted at `js/`.
- The VitaDeck-owned runtime bundle is a separate workspace package from the SDK.
- Prefer naming the VitaDeck-owned JavaScript bundle `runtime.js` instead of `main.js`.

## Package Format

The SDK writes a Deck App Package Directory under the configured output parent directory. The final package directory name is generated by the SDK, is filesystem-safe, and always ends in `.vdapp`.

Example SDK config:

```json
{
  "name": "My Deck App",
  "entry": "./src/App.tsx",
  "outDir": "./dist"
}
```

Example output:

```text
dist/
  my-deck-app.vdapp/
    manifest.json
    app.js
```

Initial package manifest:

```json
{
  "schemaVersion": 1,
  "name": "My Deck App",
  "entry": "app.js"
}
```

`app.js` is generated by the SDK. It registers the compiled Deck App Component through an internal Package Registration Hook provided by VitaDeck's runtime bundle. The hook is a compatibility convention, not a security boundary.

## Proposed Workspace Layout

```text
js/
  package.json
  pnpm-workspace.yaml
  packages/
    sdk/
      package.json
      src/
      tsconfig.deck-app.json
    runtime/
      package.json
      src/
  examples/
    hello/
      package.json
      vitadeck.config.json
      tsconfig.json
      src/App.tsx
    counters/
    timers/
    minesweeper/
```

The root `js/package.json` orchestrates workspace-level build, typecheck, lint, and example package builds. Examples should behave like standalone Deck App projects while still living in-repo for development.

## Build Flow

1. A Deck App project imports author-facing APIs from `@vitadeck/sdk`.
2. The Deck App project runs the SDK build command.
3. The SDK reads `vitadeck.config.json`, validates the project, and builds a `.vdapp` directory.
4. The SDK externalizes Runtime Dependencies such as React and SDK runtime imports.
5. VitaDeck config points at one built `.vdapp` directory.
6. The VitaDeck build copies its own runtime bundle and the selected `.vdapp` directory into the native build output.
7. Native C loads the VitaDeck runtime bundle.
8. The runtime bundle reads the package manifest, evaluates the package entry, receives the registered Deck App Component, wraps it in VitaDeck-owned providers, and renders it.

## Proposed Runtime Files

Host build output should move toward this shape:

```text
out/
  vitadeck
  js/
    runtime.js
  deck-app/
    manifest.json
    app.js
```

Vita packaging should copy the same logical files into the VPK. Exact target paths can be finalized during implementation.

## Not In Iteration 2

- Publishing `@vitadeck/sdk` to npm.
- Runtime Upload over HTTP.
- Zipped package ingestion on Vita.
- Multiple installed Deck Apps.
- App picker or VitaDeck shell UI.
- Package permissions.
- Icons, settings screens, or version metadata beyond package schema version.
- Storage, networking, or device APIs for Deck Apps.
- Strong isolation or security boundaries between Deck App code and VitaDeck globals.

## Acceptance Criteria

- A standalone package-shaped example imports from `@vitadeck/sdk` and builds a `.vdapp` directory.
- A `.vdapp` directory always contains `manifest.json` and `app.js`.
- The generated package directory name always ends in `.vdapp`.
- Deck App author code can use TypeScript and ES modules.
- Deck App author code only needs to default-export a React component.
- Deck App Packages do not bundle React.
- SDK builds validate React compatibility.
- VitaDeck selects a built `.vdapp` directory through config and copies it during native builds.
- VitaDeck loads its own runtime bundle separately from the selected Deck App Package Entry.
- Existing sample Deck Apps are represented as package-shaped examples in the workspace.
